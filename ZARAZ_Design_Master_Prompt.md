# ZARAZ — Master Prompt для Claude Design
### Повний high-fidelity редизайн UI/UX платформи бронювання послуг

> **Як користуватись цим документом:** це самодостатній технічний і творчий бриф. У ньому навмисно закрито всі рішення, які зазвичай доводиться уточнювати (кольори, шрифти, сітка, повний перелік екранів). Створюй прототип, спираючись лише на цей документ — без додаткових запитань до користувача.

---

## 0. Роль і формат відповіді

Ти — провідний продуктовий дизайнер рівня Apple / Linear / Stripe, якому доручили повністю переосмислити UI/UX реального маркетплейсу бронювань, що вже працює в проді. **Логіку, дані, ролі та функціонал НЕ можна вигадувати чи змінювати** — лише перевдягнути в новий, значно преміальніший інтерфейс, який без проблем ляже на існуючу Next.js-структуру застосунку (перелік екранів — розділ 10).

Створи повноцінний high-fidelity прототип: дизайн-систему (токени + компоненти) і ключові екрани для трьох ролей користувачів (клієнт, бізнес, супер-адмін), mobile-first там, де це релевантно (розділ 12), з усіма станами інтерактивних елементів (default / hover / focus / active / disabled / loading / empty / error).

## 1. Про продукт

**ZARAZ** — єдина платформа для онлайн-запису на будь-яку послугу: краса, здоров'я, авто, спорт, освіта, професійні послуги. Мета користувача завжди одна: **знайти послугу → знайти бізнес → обрати дату, час і (за потреби) спеціаліста → записатися за кілька кліків.**

Зараз сервіс запускається лише в м. Стрий, але дизайн має виглядати так, ніби продукт вже готовий масштабуватися на всю Україну — як національний, а не локальний сервіс.

**5-річне бачення:** ZARAZ = український Booking.com / Airbnb для сервісного бізнесу будь-якого типу — барбершопи, салони краси, манікюр, косметологія, **стоматологія, гінекологія, приватні клініки (10+ лікарів)**, автомайстерні, детейлінг, шиномонтаж, масаж, SPA, спорт, освіта, юристи, фотографи, ветеринари — і будь-яка нова категорія, яку платформа зможе додати без переробки архітектури (див. розділ 11).

**Північна зірка кожного дизайн-рішення:** *"Чи робить це пошук і бронювання швидшим, простішим і зрозумілішим?"* Якщо ні — переробити. Зручність завжди важливіша за візуальну красу.

**Емоція, яку має викликати інтерфейс** — не "вау, який красивий сайт", а:
- "Тут усе дуже зрозуміло."
- "Цьому сервісу можна довіряти."
- "Тут, мабуть, зібрані найкращі бізнеси міста."
- "Записатися буде легко."

## 2. Стильова суміш (дотримуватись пропорцій)

**60% Apple-inspired Neo-Minimalism** — чистий інтерфейс, багато повітря, відмінна типографіка, проста навігація, преміальна простота, максимум уваги контенту, нічого зайвого.

**30% Swiss Design** — чітка модульна сітка, ідеальна ієрархія та вирівнювання, великі, але функціональні відступи, логічне групування інформації, читабельність понад декоративність.

**10% Modern Luxury** — преміальність через якість виконання, а не декор: дорога типографіка (масштаб і контраст ваги, а не купа шрифтів), якісні фотографії бізнесів, дуже делікатні тіні, плавні мікроанімації, ідеальна деталізація країв/кутів/іконок.

**Орієнтири по філософії (не копіювати інтерфейси буквально):**
- **Airbnb** — довіра через прозорість і просту навігацію
- **Booking.com** — логіка пошуку, фільтрів і бронювання
- **Uber** — мінімум кроків і зайвих рішень
- **Notion** — чистота і структурованість
- **Linear** — преміальний мінімалізм, впевнена типографіка
- **Stripe** — якість типографіки, анімацій і сучасного інтерфейсу загалом

## 3. Колірна філософія (точні токени — використовувати як є)

Тепла мінімалістична світла палітра. **Це повна заміна старої темної палітри проєкту** — редизайн light-first (старий перемикач теми не є пріоритетом цього брифу; якщо додаєш dark-режим — він другорядний і будується вже після затвердження light-версії).

```
--color-bg:            #FCFBF8   /* основний фон, теплий білий */
--color-bg-secondary:  #F5F1EB   /* другорядний фон, дуже світлий беж — секції, розділювачі */
--color-surface:       #FFFFFF   /* картки — чисто білі, контраст проти теплого фону */
--color-text:          #1A1A1A   /* основний текст, майже чорний (не чистий #000) */
--color-text-muted:    #5F5F5F   /* другорядний текст, нейтральний сірий */
--color-border:        #EAE6DF   /* тонкі розділювачі — теплий світло-бежевий, не сірий */
```

**Єдиний фірмовий акцент** — один колір, що м'яко переливається фіолетовий → синій → бірюзовий (наприклад базовий тон `#6C5CE7` з можливим градієнтним нюансом у бренд-елементах на кшталт логотипу чи бейджа TOP, **але не в звичайних UI-елементах** — кнопки й посилання мають лишатись суцільним кольором, без градієнтів):

```
--color-accent:        #6C5CE7
--color-accent-hover:  #5B4BD1   /* ~10% темніше */
--color-accent-soft:   #EFEBFC   /* фон для активних чипів/бейджів, дуже світлий відтінок акценту */
```

**Правило використання акценту:** тільки для (1) головного CTA-кнопки на екрані, (2) активного стану вибору (обрана дата/час/таб), (3) посилань і фокус-стану полів, (4) індикатора завантаженості/прогресу. Ніде більше. Акцент — не декорація, а вказівник дії.

Службові кольори (мінімальне використання, тільки коли це дійсно статус):
```
--color-success: #2E9E6D    /* підтверджено, оплачено */
--color-warning: #C98A2E    /* очікує оплати, прострочено скоро */
--color-danger:  #D6493C    /* відхилено, заблоковано, помилка */
```

## 4. Типографіка

**Одна основна гарнітура на весь інтерфейс** (у дусі Linear/Apple — не змішувати шрифти заради різноманіття): **Inter** (variable font) — нейтральна, преміальна, з бездоганною підтримкою кирилиці, індустріальний стандарт для продуктів рівня Stripe/Linear.

**Другорядна display-гарнітура** — лише для великих заголовків hero-секцій і назви бізнесу на профілі (10% luxury-акценту через типографіку): **Unbounded** (геометрична, впевнена, повна підтримка кирилиці). Використовувати щонайбільше в 3-4 місцях на весь застосунок (Home hero, назва бізнесу на сторінці профілю, можливо заголовок картки TOP).

**Моно-гарнітура для чисел** — ціни, час, дати, лічильники: **JetBrains Mono** з `font-variant-numeric: tabular-nums` (щоб цифри не "стрибали" у списках).

**Масштаб (mobile → desktop через `clamp()`):**
```
H1 (hero):      clamp(32px, 6vw, 56px) / line-height 1.05 / letter-spacing -0.02em / Unbounded
H2 (section):   clamp(24px, 3vw, 34px) / line-height 1.15 / letter-spacing -0.01em / Inter 700
H3 (card title):20px / line-height 1.3 / Inter 600
Body:           16px / line-height 1.6 / Inter 400-500
Small/meta:     13-14px / line-height 1.5 / Inter 400, --color-text-muted
```

**Важливо для кирилиці:** українські слова в середньому на 20-30% довші за англійські еквіваленти. Всі кнопки, чипи фільтрів, таби навігації, бейджі статусів — проєктувати з запасом ширини і `white-space: nowrap` там, де перенос ламає композицію; ніколи не фіксувати ширину кнопки під конкретний текст.

## 5. Сітка, відступи, breakpoints

**Spacing scale (8pt база):** 4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96 px — усі відступи й gap беруться лише з цієї шкали, довільних значень немає.

**Контейнер:** max-width 1200px (desktop), padding 24px (mobile) → 40px (tablet+) → auto margin.

**Сітка:** 4 колонки (mobile) / 8 колонок (tablet) / 12 колонок (desktop), gutter 16-24px.

**Breakpoints:** 375 (mobile base) / 640 (large mobile) / 1024 (tablet/business-account) / 1280 (desktop) / 1536 (wide desktop, admin-таблиці).

**Вертикальний ритм секцій:** mobile 48-64px між секціями, desktop 96-120px. Картки всередині списку — gap 16-24px, ніколи не впритул.

## 6. Дизайн-система — компоненти (з усіма станами)

Опиши й намалюй кожен компонент мінімум у станах: default, hover, focus-visible (видиме кільце фокусу для доступності), active/pressed, disabled, loading, error.

- **Кнопки:** primary (заливка акцентом, білий текст), secondary (контур, --color-text), ghost/text (без фону, лише текст+підкреслення на hover), destructive (--color-danger, лише для незворотних дій). Радіус кутів — один і той самий для всіх кнопок в системі (напр. 12px), без варіацій.
- **Поля вводу:** текст, телефон, пароль (з іконкою показу), textarea, select, date-picker, time-slot picker (сітка чипів часу, обраний — заливка акцентом, зайнятий — приглушений і недоступний).
- **Картки:** businesses-картка в каталозі, картка послуги, картка майстра/лікаря, картка запису (booking), картка рахунку. Тінь — одна делікатна `0 1px 2px rgba(26,26,26,0.04), 0 8px 24px rgba(26,26,26,0.06)`, трохи глибша на hover для клікабельних карток (+ `translateY(-2px)`).
- **Навігація:** header (клієнтська частина, sticky, backdrop-blur на скролі), бічна навігація (business-account/admin, десктоп), нижня таб-бар навігація (client/business, мобільний).
- **Календар і вибір часу:** місячний/тижневий вигляд для бізнес-кабінету (з кольоровим кодуванням по майстру/лікарю, якщо їх декілька), простий вибір дати+часу для клієнта (горизонтальна стрічка дат + сітка слотів).
- **Повідомлення:** inline-фідбек (успіх/помилка під формою — без блокуючих `alert()`), toast для фонових подій, банер-попередження (прострочений рахунок тощо).
- **Модальні вікна:** підтвердження дії, форма (перенесення запису, ручний запис бізнесом), bottom-sheet на мобільному замість центрованого модального вікна.
- **Skeleton loading:** для кожного типу картки й списку — форма skeleton повинна повторювати реальну структуру контенту (не просто сірий прямокутник).
- **Empty states:** ілюстрація (проста лінійна, не мультяшна) + текст пояснення + CTA, якщо є дія (напр. "Немає записів — знайти послугу").
- **Бейджі статусів:** TOP-розміщення, статус запису (підтверджено/виконано/скасовано), статус рахунку (очікує/на перевірці/оплачено/прострочено) — кожен статус має один колір + іконку, ніколи лише колір без тексту (доступність).

## 7. Анімації та мікровзаємодії

Тільки мінімальні, функціональні анімації, тривалість 150-250ms, easing `cubic-bezier(0.4,0,0.2,1)`:
- Картка злегка піднімається при наведенні (2-4px) + тінь поглиблюється
- Кнопки плавно змінюють фон/тінь при hover/active, без "стрибків"
- Fade-in + невеликий translateY(8px→0) для контенту, що з'являється при скролі чи фільтрації
- Плавний перехід між станами календаря/списку результатів (крос-фейд, не миттєва заміна)
- Skeleton → контент: fade, не миттєве підміну
- Жодних анімацій "заради ефекту" — кожна пояснює зміну стану, а не розважає

## 8. Заборонено

Типовий SaaS-шаблон; Bootstrap-стилістика; надмірні градієнти на UI-елементах; важкий Glassmorphism; Neumorphism; перевантажений Hero з купою елементів; зайві ілюстрації-стокарт; кричущі анімації; візуальний шум; dark patterns; штучна терміновість (fake urgency); більше однієї яскравої акцентної точки на екран; довільні (не з палітри/шкали) кольори й відступи.

## 9. Mobile-first — але по-різному для кожної ролі

- **Клієнт** — головний пріоритет: телефон. Проєктувати спершу мобільний екран, потім масштабувати на desktop. Це основний спосіб використання ZARAZ клієнтами.
- **Бізнес-кабінет** — переважно використовується на ноутбуці/ПК адміністратором закладу. Desktop-компонування первинне (щільніші таблиці, бічна навігація, багатоколончасті форми), але мобільна версія все одно повинна бути повністю робочою і зручною (власник може заглянути з телефону).
- **Супер-адмін/модератор/фінанси** — практично завжди desktop (робота з таблицями, модерацією, фінансами). Проєктувати desktop-first, мобільна версія — компактна, але другорядна.

## 10. Повний перелік екранів і потоків (нічого не вигадувати понад це)

### Публічна частина (mobile-first)
1. **Головна** — hero з заголовком і підзаголовком, пошук (текстове поле "яку послугу шукаєш" + вибір дати), сітка популярних категорій (включно з медичними — стоматологія, гінекологія тощо), блок "популярне в місті", посилання "всі заклади".
2. **Каталог** — фільтри (категорія, район, ціна, дата), сортування (за рейтингом / за ціною), список карток бізнесів: **назва по центру картки, рейтинг, ціна від, найближчий вільний слот, TOP-бейдж якщо є — без декоративних кругових іконок біля назви**. Порожній стан, коли фільтр нічого не знайшов.
3. **Профіль бізнесу + бронювання** — обкладинка, назва, рейтинг, опис, адреса/контакти/соцмережі, список послуг (назва, тривалість, ціна, опис — послуги можуть належати до різних категорій в одному бізнесі, напр. клініка з стоматологією + оглядом зору), картки майстрів/лікарів (фото, ім'я, роль, біо), відгуки, панель бронювання (вибір послуги → дата → час → спеціаліст, якщо декілька → **необов'язкове текстове поле "коментар" від клієнта** (напр. "болить зуб зверху") → підтвердження, з нагадуванням про політику скасування).
4. **Вхід / Реєстрація** — таби "клієнт" / "бізнес" (для бізнесу — додаткові поля назви й категорії з опцією "інше" для нової категорії), чекбокс згоди з політикою конфіденційності, що блокує кнопку реєстрації, поки не позначено.
5. **Політика конфіденційності** — простий текстовий документ-сторінка.

### Клієнтський кабінет (mobile-first, бічна навігація на desktop / нижній таб-бар на мобільному)
6. **Мої записи** — таби (майбутні/минулі/скасовані), картка запису зі статусом, дії (скасувати/перенести/залишити відгук).
7. **Улюблені** — сітка збережених бізнесів.
8. **Сповіщення** — список (нагадування, статуси), непрочитані виділені.
9. **Профіль** — аватар (з можливістю завантажити і прибрати), ім'я/телефон, **зміна пароля**, статистика (виконано записів, витрачено), рейтинг клієнта, мова/тема.

### Бізнес-кабінет (desktop-first, повна мобільна адаптація)
10. **Дашборд** — ключові метрики (записи сьогодні/тиждень, дохід), рейтинг, картка просування TOP (пакети, реквізити оплати, підтвердження квитанцією, статус активації).
11. **Календар** — тижневий вигляд, кольорове кодування по майстру/лікарю, ручний запис, **можливість позначити майстру вихідний/відпустку**, деталі запису (клієнт, телефон з таймінгом розкриття, коментар клієнта, **можливість businessу відредагувати фактичну тривалість запису**).
12. **Послуги** — список з можливістю додати/редагувати (назва, опис, ціна, тривалість) / видалити, послуги можуть бути будь-якої категорії незалежно від головної категорії бізнесу.
13. **Майстри/лікарі** — список з фото (завантажити/прибрати), ім'я, роль, біо, тижневий розклад, вихідні.
14. **Відгуки** — список з можливістю відповісти.
15. **Рахунки (білінг)** — історія рахунків, статус (очікує/на перевірці/оплачено/прострочено), реквізити оплати, завантаження PDF-квитанції, **історія всіх спроб і причин відхилення**, не лише остання.
16. **Аналітика** — графіки з **вибором періоду** (не лише фіксовані 30 днів).
17. **Налаштування** — опис, адреса, телефон, соцмережі, Google Maps, обкладинка й галерея (із можливістю видалити фото без обов'язкової заміни), посилання на резервну Google-таблицю.
18. **Сповіщення (нова сторінка)** — нагадування про оплату та інші системні події в кабінеті, не лише email.

### Супер-адмін / модератор / фінанси (desktop-first)
19. **Дашборд** — загальні KPI платформи, черга нових бізнесів на розгляд із повною інформацією (опис, адреса, телефон).
20. **Бізнеси** — список зі статусом (**статус перекладений, не сирий enum**), картка деталей бізнесу (статистика, контакти власника, **прямі посилання на рахунки й TOP-заявки саме цього бізнесу**), блокування, безповоротне видалення (із захистом, якщо є майбутні підтверджені записи).
21. **Користувачі** — клієнти/власники бізнесів, пошук, блокування, видалення.
22. **Модерація відгуків** — черга на затвердження/відхилення.
23. **TOP-заявки** — черга заявок, що очікують підтвердження після оплати клієнтом, **поле причини відхилення** (аналогічно рахункам), таймер авто-активації.
24. **Категорії (нове)** — черга користувацьких категорій "інше", запропонованих бізнесами при реєстрації, з можливістю схвалити (додається до загального списку) або відхилити.
25. **Фінанси** — черга рахунків на перевірку квитанцій, підтвердження/відхилення оплати.
26. **Аналітика платформи** — графіки, розподіл за категоріями.
27. **Команда** — список модераторів/фінансистів, запрошення, пояснення різниці ролей.
28. **Журнал дій (новий)** — хто, коли, яку дію виконав (блокування, видалення тощо).

## 11. Масштабованість категорій (закладено в дизайн, не лише в код)

Категорії — не фіксований список іконок у коді дизайну, а **гнучка система**: сітка популярних категорій на головній і чипи фільтра в каталозі повинні виглядати так, ніби категорій може бути і 10, і 40 (горизонтальний скрол чипів на мобільному, обгортання на десктопі, іконка + підпис, без хардкоду точної кількості колонок). Один бізнес візуально може мати кілька тегів категорій одночасно (напр. клініка: "Стоматологія" + "Огляд зору" + "Терапевт") — передбач, як це виглядає на картці в каталозі (не більше 2-3 тегів видимо, решта — "+2" тощо).

## 12. Технічні обмеження інтеграції (не ламати логіку)

Це редизайн **лише UI/UX**, що має лягти на існуючий Next.js 14 App Router застосунок без зміни маршрутів чи бізнес-логіки:

- Маршрути: `/[locale]` (головна), `/[locale]/catalog`, `/[locale]/business/[id]`, `/[locale]/login`, `/[locale]/privacy`, `/[locale]/client/{bookings,favorites,notifications,profile}`, `/[locale]/business-account/{dashboard,settings,calendar,services,staff,reviews,billing,analytics}`, `/[locale]/admin/{dashboard,businesses,businesses/[id],users,reviews,top-placements,invoices,analytics,team}`.
- Локалізація: українська (основна) + англійська — усі текстові компоненти повинні коректно вміщати довший кириличний текст (див. розділ 4).
- Ніяких нових полів даних чи фіч понад те, що описано в розділі 10 — лише переосмислення подачі існуючого функціоналу.

## 13. Критерій готовності

Прототип вважається успішним, якщо на кожному ключовому екрані відповідь на питання **"Чи стало знайти й забронювати послугу швидше, простіше і зрозуміліше?"** — беззаперечне "так", а результат після побіжного погляду викликає одну реакцію: *"цьому сервісу можна довіряти, і тут зрозуміло, що робити."*
